Im Rahmen dieser qualifikationsrelevanten Praxisarbeit soll das Gelernte in einem praktischen Kontext zur

2 Arbeitsauftrag
Ihre Praxisarbeit soll folgende Ergebnisse liefern:
• eine lauffähige Anwendung mit Flask und einer Datenbank
• eine kurze Dokumentation zur Bedienung der Anwendung
• eine nachvollziehbare Beschreibung der Architektur der Software, insbesondere dann, wenn von den
Technologien und Vorgehensweisen aus dem Unterricht abgewichen wird.
• eine Dokumentation Architektur und der Lösungsansätze sowie die Reflexion, weshalb dies so
ausgewählt wurde, was dies für Vorteile und Nachteile bringt. Dies betrifft Fragen der Wartbarkeit,
Skalierbarkeit und Verfügbarkeit.
2.1 Anwendung
Wichtigstes Lieferobjekt ist eine lauffähige Webanwendung. Sie stellen sie zum Abgabetermin über das
Internet bereit und dort wird sie vom Examinator auf die Erfüllung der Anforderungen getestet.
Für den Zugriff über den Webbrowser und per API mit http oder https werden ein URL mit IP - Adresse
(oder DNS - Name), der Port für den Zugriff sowie die benötigten Login - Informationen dokumentiert. Die
Applikation muss nach Abgabetermin für mindestens 4 Woch en während der Korrekturzeit zur Verfügung
stehen.
ipso! Bildungsmarken
2.1.1 Art und Funktionsumfang der Webanwendung
Ihre Webapplikation stellt dem Benutzer eine Funktionalität zur Verfügung, die Sie innerhalb eines
vorgegebenen Rahmens selbst bestimmen können.
Natürlich können Sie auch die Normaufgabe (Siehe Anhang A) verwenden
Sie muss dabei folgende Anforderungen erfüllen:
• es gibt eine interaktive Weboberfläche, mit der der Benutzer Eingaben machen,
Verarbeitungen oder Aktionen auslösen und Ergebnisse ansehen kann.
• der Zugriff auf die Webanwendung muss über Benutzerkonten mit Passwörtern geregelt
werden. Neue Benutzer müssen sich mit einem eindeutigen Benutzernamen und E-Mail-
Account registrieren.
• die Applikation muss neben den Benutzeraccounts Daten speichern, die für die Applikation
spezifisch sind (Beispiel: Kochrezepte, Kommentare, Mitarbeiter, Projekte, usw…). Sie werden
in einer relationalen Datenbank gespeichert.
• es gibt eine eigene Geschäftslogik. Diese kann Berechnungen, Aktionen, Überprüfungen von
Eingaben oder auch nur die Erfassung und Anzeige von Daten umfassen.
• auf ausgewählte Daten der Webanwendung muss neben der interaktiven Weboberfläche auch
ein (mindestens) lesender Zugriff über ein RESTful Web-API mittels eines gängigen Clients
(http, curl, Postman, etc. – nicht im Browser) möglich sein. Das gilt für einen spezifischen
Inhalt der jeweiligen Webapplikation (also nicht nur die Daten der User) . Die Authentifizierung
des Clients am API muss ebenfalls ohne Browser möglich sein.
2.1.2 Technologien
Einzusetzen sind folgende Technologien:
• Datenbanksystem: MSSQL oder MySQL, MariaDB oder PostgreSQL
• Programmiersprache: Python Version ab 3.9
• Web-Framework: Flask, diverse Flask-Erweiterungen aus dem Unterricht
• Webserver: Gunicorn (standalone) oder in Kombination mit Nginx oder Apache
Der zusätzliche Einsatz weiterer Technologien ist möglich, er muss aber begründet und beschrieben
werden.
2.1.3 Source Code
Der Source Code der Anwendung wird vollständig als Repository auf GitHub, Bitbucket oder GitLab
bereitgestellt. Er wird vom Examinator stichprobenartig überprüft. Die URL und allenfalls User /
Passwort für einen lesenden Zugriff sind zu dokumentieren.
Der Source Code muss verständlich kommentiert sein, insbesondere dann, wenn der Code aus dem
Unterricht verändert oder erweitert wird oder deutlich von den Code-Beispielen aus dem Unterricht
abgewichen wird. Source Code, der aus fremden Quellen stammt, muss mit Quellenangaben im
Kommentar gekennzeichnet werden, so wie ein Zitat.
2.1.4 Tests
Tests, mit denen die Erfüllung der wichtigsten funktionalen und Qualitäts-Anforderungen überprüft
werden können, müssen dokumentiert werden. Dies können Beschreibungen für manuell ausgeführte
Tests sein oder automatisierte Tests. Für die Tests muss ein kurzes Testprotokoll erstellt werden, das
dokumentiert, welche Anforderungen bisher erfüllt wurden und welche nicht. Wichtig ist nicht, dass
alle Tests erfolgreich sind. Vielmehr soll gezeigt werden, dass die Tests definiert und durchgeführt
wurden.

3 Aufbau der Praxisarbeit
Der schriftliche Teil enthält folgende Abschnitte:
3.1 Management Summary
• Zusammenfassung die das Vorhaben, seine Ziele und die erreichten Ergebnisse beschreibt. Die
wichtigsten Punkte der Infrastruktur müssen ersichtlich sein: Was für eine Technologie / Plattform /
Cloud wurde gewählt, was ist der wichtigste Mehrwert / das grösste Risiko / was sollte hier das
Management unbedingt noch dazu wissen.
3.2 Anwendung
• Auflistung der wichtigsten Anforderungen
• Kurzbeschreibung der Funktionalität und Bedienung der Webanwendung im Browser ('User Manual')
• Kurze Dokumentation der API-Schnittstelle in der Form <Methode> <Endpunkt-URL> für jeden
Endpunkt, so dass sie mit einem gängigen API-Client (Ohne Browser) getestet werden kann. z.B. ‘http
GET https://host/api/inhalt’. Ist eine Authentisierung erforderlich, muss sie über das API möglich sein
und muss beschrieben werden.
• Beschreibung der Architektur mit Diagrammen, im Einzelnen:
o Das Datenmodell der Applikation als ERD oder Klassendiagramm, mit kurzer Beschreibung
o Die wichtigsten Abläufe in der Webanwendung als Aktivitäts-, Zustands- oder Sequenz-
Diagramm mit kurzer Beschreibung
o Allfällig verwendete zusätzliche Technologien, Strukturen und Abläufe, soweit sie nicht im
Unterricht behandelt wurden, sind mit Quellenverweis zu benennen und zu beschreiben.
o Bereitstellung der Komponenten (z.B. Applikation, Webserver, DB-Server, zusätzliche Tools)
als Diagramm mit Beschreibung.
• Ein Testprotokoll mit 6-12 ausgewählten wichtigen Testfällen. Pro Test muss eine Kurzbeschreibung,
das erwartete Ergebnis und das tatsächliche Ergebnis dokumentiert werden. Hier können auch Tests
stehen, die zum Zeitpunkt der Abgabe nicht erfolgreich waren.
Der Umfang des Dokuments, ohne Management Summary und Anhang, soll mindestens 8 und darf maximal
30 Seiten betragen. Für die korrekte Struktur des Dokuments orientieren Sie sich am Leitfaden für schriftliche
Arbeiten in der ipso Bildung.

Anhang A: Normaufgabe
1. Szenario
Eine Stadtverwaltung plant, die Mobilität in ihrem Verkehrsnetz zu verbessern und beauftragt ein IT-
Unternehmen mit der Entwicklung einer Online-Plattform zum Verleih von E-Scootern. Privatpersonen und
lokale Verleihfirmen können E-Scooter anmelden, Nutzerinnen und Nutzer können sie über eine App ausleihen,
fahren und bezahlen.
2. Ziel der Aufgabe
Erstellen Sie eine objektorientierte Analyse und ein Design nach rotem Faden, dass die folgenden
Anforderungen abbildet:
3. Funktionale Anforderungen
1. Registrierung und Authentifizierung
• Verleihanbieter registrieren sich und verwalten ihre Scooter-Flotte.
• Fahrerinnen und Fahrer legen Profile an, melden sich an und ab.
2. Scooter-Verwaltung
• Anbieter können Scooter hinzufügen, bearbeiten (Status, Standort) und entfernen.
• Jeder Scooter hat eine eindeutige ID, Akku-Status und GPS-Koordinaten.
3. Ausleihe und Rückgabe
• Nutzer:innen scannen QR-Codes, um einen Scooter zu ent- bzw. verriegeln.
• Start- und Endzeitpunkt sowie gefahrene Kilometern werden erfasst.
4. Abrechnung
• Abrechnung erfolgt minutengenau zu einem Basispreis + Fahrpreis pro Minute.
• Nutzer:innen hinterlegen Zahlungsmittel, System verarbeitet Transaktionen.
5. Nicht-funktionale Anforderungen
• Erweiterbarkeit: Einbindung weiterer Fahrzeugtypen (E-Bikes) soll leicht möglich sein.
• Performance: Die Plattform muss bis zu 500 gleichzeitige Ausleihen unterstützen.
